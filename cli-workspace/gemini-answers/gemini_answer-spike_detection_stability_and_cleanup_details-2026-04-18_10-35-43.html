Question

How to stabilize MotiveWave spike detection and cleanup low-volatility noise from the database?

Answer

Spike Detection Stability and Database Cleanup Analysis

1. Root Cause Analysis of Spike Detection Flaws

Problem 1: Baseline Drift after Real Spikes

  • Observation: On 2026-04-13 (TSLA), a real spike occurred at 05:47:55 (Price 390.02). However, in the subsequent bars (05:48:00+), the system treated 390.02 as the "normal" spot price and flagged the true market price (~347.xx) as a downward spike.
  • Cause: The StudyPriceSpike.java was using series.getClose(index - 1) (the previous bar's close) as the baseline for the current bar's detection. When a bar closes at or near a spike price, the baseline for the entire next bar is corrupted.

Problem 2: Mismatch between Detection and Reporting (0.0% Spikes)

  • Observation: Ticks with 0.0% change relative to the spot price were being flagged as spikes.
  • Cause: The study used a 3-bar average for detecting bounds (lowerBound/upperBound) but used the immediate lastBarClosePrice as the "Spot Price" for the JSON report and percentage calculation. This mismatch allowed a price to be "outside the average bounds" but "identical to the reporting baseline."

2. Technical Solution: Intra-Day Median Baseline

To solve both problems, the detection logic was moved to an Intra-Day Median Baseline.

Implementation Details:

  • Median Selection: Switched from a 3-bar average to a Median of the last 11 bars. The median is mathematically resistant to outliers; up to 5 consecutive bars can be spikes before the median departs from the true price level.
  • Intra-Day Boundary Check: The logic now strictly ignores bars from previous days to ensure that overnight gap-ups/downs do not cause false detections.

Java Code Fix (StudyPriceSpike.java):

// Baseline calculation for PM and RTH sessions
double medianPrice = getMedianCloseIntraDay(series, index, 11);
closePrice = (float) medianPrice; // Unified baseline for Bound Check and Reporting

float factor = percentFactor.getOrDefault(ticker, percentFactorDefault);
lowerBound = closePrice - closePrice * factor;
upperBound = closePrice + closePrice * factor;

Median Helper Logic:

private double getMedianCloseIntraDay(DataSeries series, int index, int count) {
    long currentDayStart = series.getStartTime(index);
    String currentDateStr = BBU.getDateStrFrom(currentDayStart);
    List<Double> prices = new ArrayList<>();

    for (int i = 1; i <= count && (index - i) >= 0; i++) {
        long barStart = series.getStartTime(index - i);
        if (!BBU.getDateStrFrom(barStart).equals(currentDateStr)) {
            break; // Stop at day boundary
        }
        prices.add((double) series.getClose(index - i));
    }
    // ... Sort and calculate median
}

3. High-Fidelity Noise Filtering

Implemented a 0.2% Hard Filter to eliminate low-volatility prints from the aggregated results.

  • Logic:
    java double absPercentChange = Math.abs(tradePrice - closePrice) / closePrice; boolean isMajorTicker = ticker.equals("SPY") || ticker.equals("QQQ"); if (!isMajorTicker && absPercentChange < 0.002) { continue; // Skip small noise }
  • Result: Tickers like TSLA, NVDA, and TSM now require a minimum of 0.2% price displacement to be recorded, while indices (SPY, QQQ) remain sensitive to smaller moves.

4. Database Integrity Cleanup

To align historical data with the new logic, 796 records were removed from the spike_mw_agg table.

Deletion Script Logic (delete_noise_spikes.py):

# Purge records that would now be caught by the 0.2% filter
stmt = delete(SpikeMWAgg).where(
    func.abs(SpikeMWAgg.price_change) <= 0.002,
    SpikeMWAgg.ticker.not_in(['SPY', 'QQQ'])
)
  • Impact: 197 TSLA records, 210 GOOGL records, and 169 TSM records (0.0% noise) were successfully purged.

4. Special Case: Corrupted Baseline Purge (2026-04-13)

After the standard noise cleanup (0.2%), some "False Spikes" remained at the 10.8% level. These were caused by the price returning to normal (~347.xx) while the system was still using the corrupted spike price (390.02) as its spot baseline.

Identification & Purge:

To resolve this, a targeted purge was executed specifically targeting the corrupted baseline value.
* Filters used:
* ticker == 'TSLA'
* t_date == '2026-04-13'
* spot_price == 390.02 (The corrupted baseline)
* target_price < 350 (The actual market price)
* Results: Successfully removed 16 aggregated and 22 raw false records while preserving the legitimate 12.1% spike.

6. Comparative Index Analysis: SPY & QQQ (Buffer: $0.50)

A specialized backtest was conducted for the major indices to verify the impact of the tighter $0.50 proximity buffer and investigate "Eventual Hit" behavior.

Key Findings:

  • Structural Pivots: For SPY and QQQ, a spike represents a structural level in the market. While the 20-day "Momentum" hit rate is lower (~61-65%), the Eventual Hit Rate remains exceptionally high (82-84%).
  • Indices vs. Stocks (TSLA):
    • TSLA: High momentum conviction (PM spikes hit fast), but if missed, the eventual recovery is less certain.
    • Indices: Higher long-term certainty. These levels act as magnets that the market tends to revisit even months later.

Recommendation:

  • Use SPY/QQQ spikes as long-term price targets or pivot zones.
  • Use TSLA spikes for tactical momentum trades within a 5-10 day horizon.
