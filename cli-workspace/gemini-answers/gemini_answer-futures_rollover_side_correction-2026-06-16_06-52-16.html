一、 交割周订单流底层机制的定位

在交割换月周，MotiveWave 连续合约图表（在 2026-06-15 显示 ESU26 约 7620 的价格带）底层的 Delta 数据实际直接来自于当前最活跃的到期合约（即 ESM26，其原生成交 Delta 在 12:55-13:00 为 -3,061，极度吻合图表显示的 -2,900左右）。而在连续图表上，历史价格被叠加了与下季度合约之间的基差（本案中对齐基差精确为 +63.75 点）。
* 二进制 Native Side 准确性：MotiveWave 本地二进制历史归档 .tick_data 文件中的 sideByte（偏移 13/14）是由交易所直接撮合成交的原始买卖方向，不受任何连续合约 Back-adjustment 干扰，是绝对准确的。因此，在二进制数据解析中绝对不应应用任何 Tick Test 纠偏，应直接提取原生 sideByte。
* 实时 Back-adjust 干扰：在交割周的实时交易中，MotiveWave 传递给实时 onTick 的成交价格会被加上基差调整（例如平移约 64 点），但 DOM 盘口价格在导出时如果不作同步调整，就会导致 tick.isAskTick() 产生 100% 被误判为 ASK 的重大偏差。

二、 系统模块的重构与升级

针对上述物理机制，我们对 Java 实时导出端 and Python 特征分析端进行了系统性的升级：
1. Python 二进制解析器：在 mw_tick_parser_backfill_raw.py 中新增了 --align-price-to 命令行参数。如果指定了对齐目标合约（如 ESU26），解析器会分别读取两合约在当天的第一个有效 Tick 价格，动态计算并四舍五入得出基差 offset，然后自适应地将主连价格加上该 offset，并将 Contract 标记为连续合约符号。同时彻底移除了 binary 解析中的错误纠偏代码。
2. Java 实时导出器：在 StudyOrderFlowDataExporter.java 中：
* 实时 onTick 端：增加了对 DOM 实时买一（Bid 0）和卖一（Ask 0）价格的追踪。在交割周内，当检测到实时 tick 价格与 DOM 中点 mid 发生由于 Back-adjust 导致的巨大偏离（如大于 1.0 点）时，动态逆向平移 tick 价格以对齐 DOM 盘口，并使用 Quote Rule 进行买卖方向的精准判定，规避了 Back-adjust 对 isAskTick() 的失真影响。
* 二进制 backfill 端：在 parseTickDataFile 中同步删除了 isRollover 纠偏，恢复对原生的 sideByte 的直接读写。
3. Python 特征生成与 Sentinel：在 order_flow_feature_generator.py 和 order_flow_sentinel.py 中，彻底移除了交割周对 CSV 原有 Side 重新进行纠偏转换的逻辑，直接信任并使用 CSV 文件中原生的、已经对齐判定好的 Side 列。

工具自动识别并计算出基差 +63.75 点，成功生成了 426,362 行无重复的干净 tick file ES_20260615_TICKS.csv。
