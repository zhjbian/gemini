Question

How should the RTH_FH (First Hour) trading hour be integrated and backfilled, and how can the MotiveWave spike detection logic be stabilized to handle baseline drift and noise?

Answer

Institutional Flow Refinement and Spike Detection Stabilization

1. RTH_FH Integration

The "First Hour" (RTH_FH) classification was refined to be inclusive of both boundaries (06:30 and 07:30 PT).
* Database Level: Updated order_flow_big_trade_add and backfillRthFh in db_query_order_flow.py to use 06:30 <= trade_time <= 07:30.
* Backtest Layer: Implemented a fallback in backtest_big_trades.py that recognizes RTH as FirstHour if it falls within that window, ensuring historical parity.
* Safety: Excluded bb_date_time.py from changes to avoid impacting other systems.

2. MotiveWave Spike Detection Refinement

The StudyPriceSpike.java study was overhauled to prevent baseline drift and reporting errors.
* Median Baseline: Shifted from a 3-bar average to a Median of the last 11 bars.
* Intra-Day Boundary: The calculation strictly ignores bars from previous days to prevent "Gap Up/Down" skewing.
* Hard Filter: Enforced a 0.2% minimum price change for all tickers except SPY and QQQ.
* Unified Signal: The reported spotPrice and price_change are now synchronized with the median baseline used for detection.

3. Historical Data Cleanup

Performed a targeted cleanup of the spike_mw_agg database table.
* Count: 796 records identified as "Noise" (<= 0.2% change).
* Action: Deleted these records for all tickers except the majors (SPY/QQQ) to improve the quality of historical signal analysis.

Files Impacted:

  • db_query_order_flow.py: RTH_FH ingestion logic.
  • backtest_big_trades.py: Backtest categorization synchronization.
  • StudyPriceSpike.java: Core spike detection logic and filters.
  • delete_noise_spikes.py (scratch): Database cleanup script.
